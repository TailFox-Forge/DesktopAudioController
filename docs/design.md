# DesktopAudioController 설계 문서

## 1. 목적

이 애플리케이션의 목적은 Windows에서 **사용자가 선택한 출력 장치만 명확하게 표시**하고,
해당 장치에 대해서만 빠르게 아래 동작을 수행하게 만드는 것입니다.

- 장치별 마스터 볼륨 조절
- 장치별 음소거
- 기본 출력 장치 변경
- 선택한 장치에서 재생 중인 프로그램 목록 확인
- 프로그램별 세션 볼륨 조절
- 프로그램별 세션 음소거
- 설정을 통한 장치 표시 목록 변경

핵심은 `모든 장치가 다 보이는 UI`가 아니라, **자주 쓰는 장치만 보이는 단순한 UI**와
**장치 선택 시 해당 장치가 실제로 출력 중인 프로그램만 펼쳐서 제어하는 구조**입니다.

### 1-1. 한눈에 보는 기능

| 기능 | 설명 | MVP 포함 여부 |
|---|---|---|
| 선택 장치만 표시 | 사용자가 고른 출력 장치만 메인 UI에 표시 | 포함 |
| 초기 설정창 | 첫 실행 시 표시할 장치 선택 | 포함 |
| 설정 재진입 | 설정창에서 나중에 장치 선택 변경 | 포함 |
| 장치 마스터 볼륨 조절 | 선택 장치의 엔드포인트 볼륨 변경 | 포함 |
| 장치별 음소거 | 장치별 mute/unmute 토글 | 포함 |
| 기본 출력 장치 변경 | 버튼으로 기본 재생 장치 전환 | 포함 |
| 장치 카드 펼치기 | 장치 선택 시 해당 장치의 세션 목록 표시 | 포함 |
| 앱 세션 목록 표시 | 해당 장치에서 재생 중인 프로그램만 표시 | 포함 |
| 앱 세션별 볼륨 조절 | 프로그램별 볼륨 슬라이더 | 포함 |
| 앱 세션별 음소거 | 프로그램별 mute/unmute 토글 | 포함 |
| 트레이 최소화 | 트레이 아이콘으로 숨김/복원 | 포함 |
| 장치 hotplug 대응 | 장치 재연결/제거 시 UI 갱신 | 포함 |
| 입력 장치 제어 | 마이크/녹음 장치 제어 | 제외 |

## 2. 범위

### 2-1. 1차 MVP 포함

1. 최초 실행 시 설정창 표시
2. 현재 활성/비활성 출력 장치 목록 조회
3. 사용자가 체크한 장치만 메인 화면에 노출
4. 선택 장치의 마스터 볼륨 조절
5. 선택 장치 음소거 토글
6. 장치 카드 선택/확장 시 해당 장치의 재생 세션 목록 조회
7. 앱 세션별 볼륨 조절 및 음소거
8. 선택 장치를 기본 출력 장치로 변경
9. 설정 저장 및 재실행 후 복원
10. 시스템 트레이 최소화

### 2-2. 1차 MVP 제외

1. 입력 장치(마이크) 제어
2. 앱별 출력 장치 라우팅 변경
3. 이퀄라이저 / DSP
4. 클라우드 동기화
5. 다국어
6. 설치 관리자 고도화

## 3. 사용자 시나리오

### 시나리오 A: 최초 실행

1. 앱을 처음 실행한다.
2. 설정창이 열린다.
3. 현재 PC의 출력 장치 목록이 표시된다.
4. 사용자는 `표시할 장치`를 체크한다.
5. 저장 후 메인창에는 선택 장치만 나타난다.

### 시나리오 B: 기본 장치 전환

1. 메인창에서 헤드셋 카드의 `기본 장치로 설정` 버튼을 누른다.
2. Windows 기본 출력 장치가 헤드셋으로 바뀐다.
3. 메인 UI에 현재 기본 장치 상태가 반영된다.

### 시나리오 C: 장치 확장 후 프로그램별 볼륨 조절

1. 사용자가 스피커 장치 카드를 클릭한다.
2. 장치 카드가 펼쳐지고 해당 장치에서 실제 출력 중인 프로그램 목록이 보인다.
3. 예를 들어 `Chrome`, `Discord`, `Game.exe` 같은 세션이 표시된다.
4. 사용자는 `Chrome` 세션 볼륨만 낮춘다.
5. 장치 마스터 볼륨은 유지된 채, 해당 프로그램 볼륨만 조정된다.

### 시나리오 D: 설정 변경

1. 트레이 메뉴나 메인 UI에서 설정창을 연다.
2. 보이지 않던 장치를 새로 체크한다.
3. 저장 후 메인창 목록이 즉시 갱신된다.

## 4. 기술 선택

## 4-1. UI 프레임워크

선택:

```text
WPF
```

선정 이유:

- Windows 시스템 유틸리티에 적합
- 트레이, 설정창, COM 연동이 비교적 단순
- WinUI 3보다 MVP 구현 마찰이 적음

WinUI 3를 당장 쓰지 않는 이유:

- 최신 UI 장점은 있으나
- 트레이 기반 유틸리티 / COM 오디오 제어 / 배포 단순성에서는 WPF가 유리

## 4-2. 오디오 제어 계층

권장 조합:

```text
- 장치 열거 / 볼륨 조회 / 음소거:
  NAudio.CoreAudioApi 또는 동등한 Core Audio wrapper

- 세션 열거 / 세션 볼륨 / 세션 음소거:
  IAudioSessionManager2 계층 접근 또는 동등 wrapper

- 기본 출력 장치 전환:
  Windows PolicyConfig COM interop
```

이 조합을 쓰는 이유:

- 장치 목록/볼륨 쪽은 NAudio가 생산성이 좋음
- 앱 세션 제어는 장치 제어와 다른 계층이므로 별도 세션 서비스가 필요
- 기본 장치 전환은 Core Audio 표준 wrapper만으로는 부족한 경우가 있어 별도 COM interop 필요

## 5. 아키텍처

단순한 MVVM 구조로 시작합니다.

```text
UI
- Views
  - MainWindow
  - SettingsWindow

- ViewModels
  - MainViewModel
  - SettingsViewModel
  - DeviceCardViewModel
  - AudioSessionItemViewModel

Application
- Services
  - AudioDeviceService
  - AudioSessionService
  - DefaultDeviceService
  - DeviceWatcherService
  - SettingsService
  - TrayService

Domain
- Models
  - AudioDeviceInfo
  - AudioSessionInfo
  - AppSettings
```

### 5-1. 역할

#### AudioDeviceService

- 출력 장치 목록 조회
- 장치 볼륨 읽기/쓰기
- 장치 음소거 읽기/쓰기

#### AudioSessionService

- 특정 장치에서 활성인 오디오 세션 목록 조회
- 세션 볼륨 읽기/쓰기
- 세션 음소거 읽기/쓰기
- 프로세스명 / 표시명 / 아이콘 식별 지원

#### DefaultDeviceService

- 특정 출력 장치를 Windows 기본 장치로 설정
- 현재 기본 장치 조회

#### DeviceWatcherService

- 장치 추가/제거/상태 변경 이벤트 감지
- UI 갱신 이벤트 발행

#### SettingsService

- JSON 설정 파일 읽기/쓰기
- visible device 목록 관리
- UI 상태 복원

#### TrayService

- 시스템 트레이 아이콘
- 메인창 숨김/복원
- 설정창 열기/종료 메뉴 제공

```text

```

설정 예시:

```json
{
  "visibleDeviceIds": [
    "device-id-1",
    "device-id-2"
  ],
  "window": {
    "left": 120,
    "top": 80,

    "height": 540

  "showOnlyConnectedDevices": true

```

설계 원칙:

- 장치명은 바뀔 수 있으므로 **ID 기반 저장**

## 8. 장치 식별과 변경 대응

문제:

- USB 헤드셋, 모니터, 블루투스 스피커는 자주 붙었다 떨어질 수 있음
- 장치명이 같아도 endpoint id가 다를 수 있음

대응:

1. 설정은 endpoint id 기준 저장
2. 장치 미연결 상태면 숨기거나 `오프라인` 표시
3. 재연결 시 동일 id면 자동 복원
4. 완전히 다른 id로 재생성되면 사용자가 설정에서 재선택
5. 앱 세션은 프로세스 종료/재생 중지 시 목록에서 자동 제거
6. 같은 프로그램이라도 세션이 여러 개면 별도 세션으로 표시될 수 있음

## 9. 예외/엣지 케이스

1. 선택된 장치가 모두 제거됨
   - 메인창에 `선택된 장치가 연결되어 있지 않습니다` 표시

2. 기본 장치 변경 실패
   - 관리자 권한/정책/드라이버 상태에 따라 실패 가능
   - 토스트 또는 메시지 표시

3. 특정 장치가 볼륨 제어 불가
   - 슬라이더 비활성화
   - 이유 메시지 표시

4. 특정 앱 세션이 짧게 생성됐다가 사라짐
   - 세션 목록을 즉시 갱신
   - 사라진 세션 UI는 자연스럽게 제거

5. 설정 파일 손상
   - 기본 설정으로 복구
   - 손상 파일 백업 후 새 파일 생성

## 11. 배포 전략

결론부터 말하면:

```text
MVP/내부 테스트: zip 배포로 충분
정식 내부 배포: installer 추가 권장
```

### 11-1. zip 배포가 가능한 조건

이 앱은 아래 성격을 가지므로, 1차 배포는 비교적 단순합니다.

- Windows 데스크톱 앱
- 별도 드라이버 없음
- 별도 서비스 설치 없음
- 로컬 설정 파일 중심

따라서 **초기 테스트용은 zip 배포**가 충분합니다.

다만 여기서 말하는 zip 배포는 보통 아래 둘 중 하나입니다.

1. **Self-contained single-file publish 결과물**
   - exe 1개 중심 배포 가능
2. **publish 폴더 전체 zip**
   - exe + 필요한 runtime 파일 포함

즉, 무조건 `exe 하나만` 던지는 방식보다, **publish 결과 전체를 zip으로 배포**하는 쪽이 더 안전합니다.

### 11-2. installer가 필요한 시점

아래 요구가 생기면 installer를 붙이는 게 맞습니다.

- 시작 메뉴 / 바탕화면 바로가기
- 자동 실행 등록
- 제거(Uninstall) 절차
- 버전 업데이트 관리
- 회사 내부 표준 배포 체계 연동

### 11-3. 권장 배포 순서

1. **1차 MVP**
   - self-contained x64 publish
   - zip 배포

2. **2차 내부 확산**
   - installer 추가
   - 자동 실행 / 바로가기 / 제거 포함

3. **3차 정식 배포**
   - 버전 업그레이드 정책
   - 서명(Signing) 여부 검토

### 11-4. 현재 권장안

현재 프로젝트 단계에서는 아래가 적절합니다.

```text
- 구현 초기: zip 배포
- 사용자 수가 늘거나 운영 편의가 중요해지면 installer 추가
```

즉, 지금 당장은 **설치 파일이 필수는 아니고**,  
MVP 기준으로는 **zip 배포가 가장 빠르고 현실적**합니다.

## 12. 테스트 전략

### 12-1. 1차 수동 테스트

1. 첫 실행 시 설정창 열림
2. 장치 선택 후 메인 UI에 선택 장치만 노출
3. 장치 마스터 볼륨 변경 시 실제 장치 볼륨 반영
4. 장치 음소거 토글 반영
5. 장치 카드 확장 시 해당 장치의 앱 세션 목록 표시
6. 앱 세션 볼륨 변경 시 실제 프로그램 볼륨 반영
7. 기본 장치 변경 반영
8. 앱 재실행 후 설정 복원
9. 장치 제거/재연결 시 갱신

### 12-2. 자동 테스트 우선순위

- SettingsService JSON 직렬화/역직렬화
- ViewModel 상태 전이
- 장치 필터링 로직
- 세션 필터링 및 장치-세션 매핑 로직

하드웨어 종속성이 큰 오디오 제어 자체는 추상화 뒤에 두고 단위 테스트 범위를 제한합니다.

